# Discussion Log Generator

현재 대화의 논의 내용을 구조화된 discussion-log.md로 정리한다.

## 핵심 원칙

1. **Raw 메시지 보존**: 유저의 실제 발화를 `>` 인용(blockquote)으로 포함하여 뉘앙스와 맥락이 전달되도록 한다. 정리한다고 원문을 요약/변환하지 않는다.
2. **맥락 변질 방지**: 요약 과정에서 빠뜨리거나 의미가 바뀌는 것을 주의한다. 논의의 흐름(A를 주장 → B가 반론 → C로 수정)이 그대로 드러나야 한다.
3. **솔직한 반성 포함**: Claude가 너무 수용적이었거나, 잘못된 근거를 댔거나, 과도한 스펙을 작성한 경우 등을 솔직하게 기록한다.
4. **기록 > 정리**: 깔끔한 문서보다 맥락이 살아있는 문서가 목표. 의사결정의 "과정"이 보여야 한다 — 최종 결론만이 아니라 어떤 대안이 있었고 왜 기각됐는지.

## 출력 위치 결정

아래 순서로 판단한다:

1. **wizard-planning 논의였을 때**: `.wizard/output/{project}/planning/discussion-log.md`
2. **wizard-design 논의였을 때**: `.wizard/output/{project}/design/discussion-log.md`
3. **이미 discussion-log.md가 존재할 때**: 기존 파일을 Read하고, 기존 내용을 유지한 채 새 논의 내용을 아래에 append한다.
4. **그 외**: 유저에게 저장 위치를 물어본다. 후보를 2~3개 제안한다.

위치를 자동 결정할 수 없으면, AskUserQuestion으로 후보를 제안하고 선택을 받는다.

## 출력 구조

```markdown
# Discussion Log

> **기간**: {대화 시작 ~ 현재 날짜}
> **참여**: {참여자 역할}

---

## 1. {논의 토픽 제목}

### 배경
{이 논의가 시작된 맥락. 어떤 Phase에서, 어떤 작업 중에 발생했는지.}

### 논의 과정
{시간순으로 주요 발언과 반론을 기록.}

> "유저의 실제 발화를 그대로 인용"

{Claude의 대응, 리서치 결과, 제안 등}

> "유저의 후속 챌린지나 피드백"

{Claude의 수정/반성}

### 결론
{최종 의사결정과 근거. 기각된 대안이 있으면 함께 기록.}

### 교훈 (있을 경우)
{Claude가 이 논의에서 배운 것, 반성할 점. 없으면 이 섹션 생략.}

---

## 2. {다음 논의 토픽}
{같은 구조 반복}

---

## 핵심 교훈 (Claude 자체 회고)
{전체 대화에서 Claude가 개선해야 할 패턴들. 구체적 사례와 함께.}
```

## 기존 파일 업데이트 시

discussion-log.md가 이미 존재하면:
1. 기존 파일을 Read한다
2. 기존 내용은 그대로 유지한다
3. 마지막 토픽 번호를 확인하고 이어서 번호를 매긴다
4. "핵심 교훈" 섹션은 기존 + 신규를 머지한다
5. 헤더의 기간을 업데이트한다

## 인자 ($ARGUMENTS)

- **인자 있을 때**: 해당 토픽/키워드에 관련된 논의만 집중 정리 (예: `/discussion-log 세션 구조`)
- **인자 없을 때**: 전체 대화를 포괄적으로 정리

## 주의사항

- 이 스킬은 대화 컨텍스트를 기반으로 동작하므로, 대화가 길수록 더 풍부한 로그가 생성됨
- 사소한 대화(인사, 단순 확인)는 포함하지 않음 — 의사결정이나 논의가 있었던 부분만
- 유저가 챌린지한 내용은 반드시 원문 인용과 함께 포함
